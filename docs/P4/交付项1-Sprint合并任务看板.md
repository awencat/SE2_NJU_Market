# 交付项 1：Sprint 合并任务看板

## 1. 规划依据

本阶段按 `P4-编码开发.md` 的要求，将 P1 需求分析和 P3 详细设计中的功能合并为一个开发 Sprint。优先级定义如下：

| 优先级 | 定义 | 本阶段处理原则 |
| --- | --- | --- |
| P0 | 最小可用版本必须具备的核心流程 | 必须完成开发、联调和演示验证 |
| P1 | 支撑完整体验和质量验收的重要功能 | 完成基本开发与核心路径验证 |
| P2 | 增强体验或管理效率的扩展功能 | 在不影响 P0/P1 稳定性的前提下完成或延期 |
| 测试/工程 | 测试、CI、文档和演示准备 | 与功能开发同步推进 |

## 4. AI 辅助规划与人工确认

| 项目 | AI 辅助内容 | 人工确认结果 |
| --- | --- | --- |
| 任务拆解 | 根据 P4 模板将核心功能拆成 2-4 小时粒度任务 | 团队按现有代码结构调整为用户、商品、订单、评价、管理端和测试工程任务 |
| 工时估算 | 给出每项 2-4 小时的初始估算 | 团队结合成员熟悉度修正，核心订单和联调任务保留 4 小时 |
| 风险识别 | 提醒认证、订单状态、重复评价和 CI 锁文件一致性风险 | 团队将订单状态、评分边界和 `npm ci` 纳入测试/CI 验证 |
| 范围取舍 | 建议 P0 全量完成，P2 不阻塞交付 | 智能匹配延期；完整通知中心降级为订单通知和个人页展示 |

## 5. 风险与处理

| 风险 | 影响 | 处理结果 |
| --- | --- | --- |
| 前后端字段不一致 | 商品详情、订购和评价页面可能无法串联 | 通过集成测试和演示路径人工联调验证 |
| 订单状态和库存状态不一致 | 购买或取消后商品可售状态错误 | 补充服务层单元测试覆盖购买、取消、库存恢复 |
| 评分重复提交或未购买评分 | 评价体系可信度下降 | 服务层加入业务校验，并在集成测试中覆盖重复评价异常 |
| `package-lock.json` 与 `package.json` 不同步 | CI 中 `npm ci` 会失败 | 执行 `npm install` 同步锁文件，并复跑 `npm ci` 通过 |
| 前端 bundle 偏大 | 构建有警告，首屏加载可能变慢 | 本阶段不阻塞验收，后续可用路由懒加载和代码拆分优化 |
