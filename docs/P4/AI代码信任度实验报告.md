# AI 代码信任度实验报告

## 一、实验目的

本次实验用于观察 AI 在项目代码维护中的可信程度，重点关注两类任务：

1. 编译错误、字段不一致等确定性问题。
2. 涉及真实业务语义的软件功能补全问题。

实验结论倾向于：AI 对编译 BUG 和局部代码错误有较高处理效率，但对于具体业务功能的完整实现，仍需要人工明确需求、检查逻辑边界并介入修正。

## 二、实验案例一：`user.status` 字段内容不一致修复

### Prompt

项目中用户状态字段 `user.status` 的取值存在不一致问题，请检查相关代码并修复，使状态字段在不同接口、实体和业务逻辑中保持一致。

### AI 直出结果

AI 能够较快定位到状态字段相关位置，包括：

- `User` 实体中的 `status` 字段。
- 用户状态更新相关服务逻辑。
- 前后端可能使用的状态值。

AI 初步给出的方案是统一状态字段取值，并在服务层更新用户状态时使用同一组字符串常量，避免不同模块使用不同含义或不同格式的状态值。

### 人工检查问题

人工检查后发现，仅靠 AI 直出结果仍存在风险：

- AI 能发现字段不一致，但不一定知道项目中各状态值的业务含义。
- 如果没有人工确认，可能会把历史数据、前端展示值、后端判断值混为一类。
- 状态字段不仅影响编译，还会影响登录、封禁、展示、管理等业务路径。

### 修复前后对比

修复前：

- `user.status` 的取值约定不清晰。
- 不同位置可能使用不同字符串表示同一状态。
- 后续开发人员难以判断哪些状态值是合法值。

修复后：

- 统一了 `user.status` 的使用方式。
- 状态更新逻辑集中在服务层处理。
- 降低了因字符串不一致导致的判断错误。

## 三、实验案例二：`cancelOrder` 退订取消逻辑修复

### Prompt

前端 `api.js` 中已有 `cancelOrder(orderId, buyerId)` 方法。现在要求：当订单被退订取消时，后端要删除这个订单，并把该订单对应商品的 `status` 恢复为 `ON_SALE`。目前后端没有完整实现，请补全。

### AI 直出结果

AI 首先检索了前端和后端现有实现，发现：

- 前端调用路径为 `POST /api/orders/{orderId}/cancel`。
- 后端已有 `OrderEntityController.cancel(...)`。
- 后端已有 `OrderEntityServiceImpl.cancel(...)`，但原逻辑只是把订单状态更新为 `CANCELLED`。
- 商品和订单通过 `order_item` 表关联。

AI 随后修改后端取消逻辑，使其在事务中完成：

- 校验订单存在。
- 校验传入的 `buyerId` 是否为订单买家。
- 查询该订单对应的 `OrderItem`。
- 将对应商品 `status` 恢复为 `ON_SALE`。
- 删除订单项。
- 删除订单本身。

### 人工检查问题

人工介入后确认，AI 初步处理方向正确，但该任务不是单纯编译错误，而是业务语义修复，需要人工明确以下点：

- “取消订单”不是把订单状态改为 `CANCELLED`，而是物理删除订单。
- 删除订单前必须先处理 `order_item`，否则可能触发外键约束问题。
- 商品状态必须恢复为项目约定的 `ON_SALE`，不能使用前端旧值或其他状态名。
- 该逻辑应放入事务中，避免出现“订单删了但商品未恢复”或“商品恢复但订单还存在”的中间状态。

### 修复前后对比

修复前：

```text
cancelOrder -> 后端将订单 status 改为 CANCELLED
             -> 商品 status 仅在部分情况下恢复
             -> 订单记录仍保留
```

存在问题：

- 不符合“退订取消时删除订单”的业务要求。
- 订单项仍可能保留。
- 后续订单列表可能仍看到已取消订单。

修复后：

```text
cancelOrder -> 校验订单与买家
             -> 查询订单项
             -> 商品 status 恢复为 ON_SALE
             -> 删除 order_item
             -> 删除 order
```

改进效果：

- 行为符合退订取消需求。
- 商品重新回到可售状态。
- 避免订单和订单项残留。
- 使用事务保证一致性。

## 四、实验结论

本次实验说明：

1. AI 适合处理编译错误、接口路径检索、字段引用检查等确定性任务。
2. AI 能快速找到相关代码位置，并给出可运行的初步修复方案。
3. 对于业务规则较强的功能，AI 需要人工提供明确语义，例如“取消订单是删除记录，而不是改状态”。
4. 人工检查仍是必要步骤，尤其要检查数据一致性、状态流转、外键约束和事务边界。

因此，在本项目中较合理的协作方式是：

```text
人工明确业务规则 -> AI 快速定位和生成代码 -> 人工审查逻辑边界 -> AI 辅助修正和验证
```

AI 可以显著提高开发效率，但不应直接替代开发者对业务语义和系统一致性的判断。
