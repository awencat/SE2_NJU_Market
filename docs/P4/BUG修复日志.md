# BUG 修复日志

## 1. `user.status` 字段内容不一致

### 问题描述

项目中 `user.status` 字段在不同位置的内容存在不一致，导致部分用户状态判断和显示结果不稳定。

### 定位方式

该问题由人工修复。修复时主要依据运行日志输出和相关接口返回内容进行校对，检查前后端对 `user.status` 字段的使用是否一致。

### 修复过程

修复过程较为简单：

1. 根据日志输出确认异常状态值。
2. 对照用户实体、接口返回值和前端使用位置。
3. 将不一致的状态内容调整为统一约定。
4. 重新运行相关功能，确认状态显示和判断恢复正常。

### 修复结果

修复后，`user.status` 字段在相关功能中的含义和取值保持一致，减少了因字段内容不统一导致的判断错误。

## 2. `cancelOrder` 取消订单逻辑缺失

### 问题描述

前端 `api.js` 中已有 `cancelOrder(orderId, buyerId)` 方法，会调用：

```text
POST /api/orders/{orderId}/cancel
```

但后端原有实现只将订单状态改为 `CANCELLED`，没有按照需求删除订单，也没有完整清理订单项并恢复商品状态。

需求要求：

1. 订单取消时删除订单。
2. 删除订单对应的 `order_item`。
3. 将对应商品的 `status` 恢复为 `ON_SALE`。

### 定位方式

该问题由 AI 修复。修复时检索了以下代码：

- `frontend/src/api.js`
- `OrderEntityController`
- `OrderEntityService`
- `OrderEntityServiceImpl`
- `OrderEntity`
- `OrderItem`
- `Good`

确认前端调用路径已存在，问题集中在后端 `OrderEntityServiceImpl.cancel(...)` 的业务逻辑不完整。

### 修复过程

我的修复做法如下：

1. 保留原有接口路径 `POST /api/orders/{orderId}/cancel`，避免前端改动。
2. 在 `cancel(orderId, buyerId)` 中先查询订单。
3. 校验订单是否存在。
4. 如果传入 `buyerId`，校验是否为该订单买家。
5. 查询该订单对应的所有 `OrderItem`。
6. 根据订单项找到对应商品，并将商品状态恢复为 `ON_SALE`。
7. 删除该订单对应的 `order_item` 记录。
8. 删除订单记录本身。
9. 使用事务保证商品状态恢复、订单项删除和订单删除保持一致。

### 修复前后对比

修复前：

```text
cancelOrder
  -> order.status = CANCELLED
  -> 订单仍保留
  -> 订单项可能仍保留
  -> 商品状态恢复不完整
```

修复后：

```text
cancelOrder
  -> 校验订单和买家
  -> 查询订单项
  -> 商品 status 恢复为 ON_SALE
  -> 删除 order_item
  -> 删除 order
```

### 修复结果

后端取消订单逻辑已按需求改为物理删除订单，并恢复商品可售状态。前端原有 `cancelOrder` 方法无需修改。

## 3. 商品图片显示 URL 错误

### 问题描述

商品图片上传后，图片实际路径与前端显示时拼接的 URL 不一致，导致图片文件存在但页面无法正常显示。

### 定位方式

该问题由人工修复。修复时主要依据日志输出、接口返回结果和浏览器中图片加载地址进行校对。

### 修复过程

修复过程较为简单：

1. 查看商品图片上传后的接口返回路径。
2. 对照浏览器中图片实际请求 URL。
3. 发现前端仍按旧路径拼接图片地址。
4. 将商品图片显示逻辑调整为匹配当前后端静态资源路径。

### 修复前后对比

修复前：

```text
后端返回 /uploads/goods/xxx.png
前端按旧路径拼接图片 URL
图片加载失败
```

修复后：

```text
后端返回 /uploads/goods/xxx.png
前端按 /uploads/goods 路径显示图片
图片正常加载
```

### 修复结果

商品图片 URL 显示逻辑已与后端静态资源路径保持一致，图片可以正常展示。

## 总结

本次修复包含两个人工修复问题和一个 AI 辅助修复问题：

| 问题 | 修复方式 | 主要依据 |
| --- | --- | --- |
| `user.status` 字段内容不一致 | 人工修复 | 日志输出与字段内容校对 |
| `cancelOrder` 取消订单逻辑缺失 | AI 修复 | 前后端调用链检索与业务逻辑补全 |
| 商品图片显示 URL 错误 | 人工修复 | 日志输出、接口返回和图片请求地址校对 |

整体来看，日志和接口返回内容对人工快速校对问题很有帮助；AI 更适合在已明确需求后，快速补全跨实体、跨服务的后端业务流程。
