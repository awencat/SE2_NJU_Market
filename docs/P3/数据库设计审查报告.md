## 二、数据类型选择审查

### 2.1 主键与 ID 字段

| 表 | 字段 | 当前类型 | 评估 | 建议 |
|----|------|----------|------|------|
| 全部 | `*_id` | `INT AUTO_INCREMENT` | 对于校园级应用足够，约 21 亿上限 | **当前可接受**。若预期用户量级达百万以上，建议升级为 `BIGINT` |

### 2.2 user / admin 表

| 字段 | 当前类型 | 评估 | 建议 |
|------|----------|------|------|
| `username` | `VARCHAR(100)` | 实际用户名限制为 3-20 位（InputSanitizer 校验），100 略大但不浪费显著 | ✅ 可接受 |
| `password` | `VARCHAR(255)` | BCrypt 哈希输出约 60 字符，255 完全够用 | ✅ 合理 |
| `email` | `VARCHAR(120)` | RFC 5321 规定邮箱最长 254 字符，120 可能截断长邮箱 | ⚠️ 建议改为 `VARCHAR(255)` |
| `phone` | `VARCHAR(30)` | 支持国际号码格式 | ✅ 合理 |
| `avatar_url` | `VARCHAR(255)` | 普通 URL 足够，但 CDN 签名 URL 可能超长 | ⚠️ 建议改为 `VARCHAR(512)` |
| `campus` | `VARCHAR(100)` | 校名长度足够 | ✅ 合理 |
| `address` | `VARCHAR(255)` | 地址通常够用 | ✅ 合理 |
| `role`（admin）| `VARCHAR(50)` | 实际只有 `SUPER_ADMIN` 和 `CONTENT_ADMIN` 两种 | ⚠️ 建议使用 `ENUM('SUPER_ADMIN','CONTENT_ADMIN')` 约束 |

### 2.3 good 表

| 字段 | 当前类型 | 评估 | 建议 |
|------|----------|------|------|
| `title` | `VARCHAR(150)` | 商品标题，150 足够 | ✅ 合理 |
| `description` | `TEXT` | 最大 64KB，适合长文本 | ✅ 合理 |
| `price` | `DECIMAL(10,2)` | **金融计算首选定点数**，范围 ±99,999,999.99，校园交易完全够用 | ✅ 正确选择 |
| `category` | `VARCHAR(100)` | 实际取值 book/digital/daily/sports，用 VARCHAR 灵活但无约束 | ⚠️ 建议 `ENUM('book','digital','daily','sports','other')` |
| `status` | `VARCHAR(50)` | 取值 ON_SALE/SOLD，无约束 | ⚠️ 建议 `ENUM('ON_SALE','SOLD','OFF_SHELF')` |
| `condition` | `VARCHAR(50)` | 取值 USED_GOOD/USED_FAIR | ⚠️ 建议 `ENUM('USED_GOOD','USED_FAIR','USED_POOR')` |
| `view_count` | `INT DEFAULT 0` | 浏览次数计数器 | ✅ 合理 |

**关于 ENUM vs VARCHAR 的选择：**  
- **ENUM 优点**：数据完整性约束、存储效率（1-2 字节）、自文档化  
- **VARCHAR 优点**：灵活扩展，添加新类型不需 ALTER TABLE  
- **建议**：当前阶段类目/状态取值已固定，使用 ENUM 可防止脏数据；未来如需动态扩展，可配合应用层枚举一起迁移

### 2.6 时间字段：TIMESTAMP vs DATETIME

当前所有时间字段统一使用 `TIMESTAMP`：

| 特点 | TIMESTAMP | DATETIME |
|------|-----------|----------|
| 范围 | 1970-01-01 ~ 2038-01-19 | 1000-01-01 ~ 9999-12-31 |
| 时区 | 自动转换 UTC | 存储字面值 |
| 存储 | 4 字节 | 8 字节（MySQL 5.6+） |

**评估**：对于校园项目，TIMESTAMP 足够且有时区自动转换优势。但需注意 **2038 年问题**。长期维护建议统一使用 `DATETIME`。

## 三、索引设计审查

### 3.1 当前索引情况

当前 `schema.sql` 中显式定义的索引**仅有一处**：

| 表 | 索引 | 类型 | SQL |
|----|------|------|-----|
| `order` | `order_number` | UNIQUE | `order_number VARCHAR(100) NOT NULL UNIQUE` |

MySQL InnoDB 引擎会自动为以下字段创建索引：
- 所有 `PRIMARY KEY`（聚簇索引）
- 所有 `FOREIGN KEY` 约束列（非聚簇索引）

**当前 FK 自动索引覆盖的字段**：
- `good.seller_id`（FK → user）
- `order.buyer_id`、`order.seller_id`（FK → user）
- `report.reporter_id`、`report.target_user_id`、`report.target_good_id`、`report.handler_admin_id`
- `comment.user_id`、`comment.good_id`、`comment.parent_comment_id`
- `rating.user_id`、`rating.good_id`
- `favorite.user_id`、`favorite.good_id`
- `good_image.good_id`
- `order_item.order_id`、`order_item.good_id`

### 3.2 缺失的关键索引

| 优先级 | 表 | 索引字段 | 索引类型 | 原因 |
|:------:|----|----------|----------|------|
| 🔴 高 | `user` | `username` | **UNIQUE** | 登录查询的核心字段；注册时需唯一性校验。当前无此索引，每次登录都全表扫描 |
| 🔴 高 | `user` | `email` | **UNIQUE** | 注册去重、邮箱登录、找回密码等功能依赖 |
| 🟡 中 | `good` | `(category, status)` | 复合索引 | 浏览商品页最频繁的查询：`WHERE category = ? AND status = 'ON_SALE'` |
| 🟡 中 | `good` | `(status, created_at)` | 复合索引 | 首页最新商品列表：`WHERE status = 'ON_SALE' ORDER BY created_at DESC` |
| 🟡 中 | `good` | `seller_id` | （FK 已自动创建） | 查看"我发布的商品"，但 InnoDB FK 已覆盖 ✅ |
| 🟡 中 | `order` | `(buyer_id, status)` | 复合索引 | 买家查"我的订单"并按状态筛选 |
| 🟡 中 | `order` | `(seller_id, status)` | 复合索引 | 卖家查"卖出的订单"并按状态筛选 |
| 🟡 中 | `comment` | `(good_id, created_at)` | 复合索引 | 查看某商品的所有评论，按时间排序 |
| 🟡 中 | `report` | `(status, created_at)` | 复合索引 | 管理员查看待处理举报列表 |
| 🟢 低 | `favorite` | `(user_id, good_id)` | **UNIQUE** 复合索引 | 防止同一用户重复收藏同一商品 |
| 🟢 低 | `rating` | `(user_id, good_id)` | **UNIQUE** 复合索引 | 防止同一用户对同一商品重复评分 |

### 3.3 索引设计汇总 SQL

```sql
-- === 高优先级 ===

-- 登录 / 注册核心索引
ALTER TABLE `user` ADD UNIQUE INDEX idx_user_username (username);
ALTER TABLE `user` ADD UNIQUE INDEX idx_user_email (email);

-- === 中优先级 ===

-- 商品浏览：按分类 + 状态筛选（最频繁查询）
ALTER TABLE good ADD INDEX idx_good_category_status (category, status);

-- 商品浏览：首页最新在售商品
ALTER TABLE good ADD INDEX idx_good_status_created (status, created_at);

-- 买家/卖家订单列表 + 状态筛选
ALTER TABLE `order` ADD INDEX idx_order_buyer_status (buyer_id, status);
ALTER TABLE `order` ADD INDEX idx_order_seller_status (seller_id, status);

-- 商品详情页评论列表
ALTER TABLE comment ADD INDEX idx_comment_good_created (good_id, created_at);

-- 管理员：待处理举报列表
ALTER TABLE report ADD INDEX idx_report_status_created (status, created_at);

-- === 低优先级 ===

-- 防重复收藏
ALTER TABLE favorite ADD UNIQUE INDEX idx_favorite_user_good (user_id, good_id);

-- 防重复评分
ALTER TABLE rating ADD UNIQUE INDEX idx_rating_user_good (user_id, good_id);
```

### 3.4 索引设计原则回顾

本设计遵循以下原则：

1. **最左前缀原则**：复合索引 `(category, status)` 可覆盖单独查 `category` 的查询，但需注意字段顺序
2. **高选择性优先**：`username`、`email` 选择性极高（几乎唯一），适合建索引；`status` 选择性较低（仅 3-4 种取值），通常放在复合索引的后面
3. **覆盖索引**：高频查询尽量让索引覆盖所需字段，避免回表
4. **避免过度索引**：每个索引都会增加写入成本（INSERT/UPDATE/DELETE 需维护索引），需权衡读写比例

## 四、隐私数据处理审查

### 4.1 密码存储

#### 现状分析

| 层面 | 现状 | 评估 |
|------|------|------|
| **DDL（schema.sql）** | `password VARCHAR(255)` | ✅ 字段长度能容纳 BCrypt 哈希（~60 字符） |
| **种子数据（data.sql）** | 明文密码 `'123456'` | ❌ 严重问题 |
| **后端 UserServiceImpl** | 注册时 `BCryptPasswordEncoder.encode()`，登录时 `matches()` 验证 | ✅ 正确实现 |
| **后端 SecurityConfig** | 注入 `BCryptPasswordEncoder` Bean | ✅ 正确配置 |
| **User 实体** | `@JsonIgnore` 标注 password | ✅ 不会序列化到前端响应 |
| **防重复加密** | `normalizeUser()` 检测 `$2` 前缀跳过已加密密码 | ✅ 防御性设计 |

#### 问题

1. **种子数据密码为明文**：`data.sql` 中所有密码均为 `'123456'`，与 BCrypt 设计矛盾
   - **影响**：运行种子数据后，登录验证会失败（数据库存明文，代码期望 BCrypt 哈希）
   - **修复**：将种子数据中的密码替换为预生成的 BCrypt 哈希值

2. **Admin 密码双重缺失**：
   - [Admin.java](file:///d:/软工二/SE2_NJU_Market/backend/src/main/java/com/marketback/main/entity/Admin.java) 的 `password` 字段**无 `@JsonIgnore`**，存在密码泄露到前端响应的风险
   - [AdminServiceImpl](file:///d:/软工二/SE2_NJU_Market/backend/src/main/java/com/marketback/main/service/impl/AdminServiceImpl.java) 是空实现，**未对管理员密码做 BCrypt 加密**

#### 修复建议

```sql
-- 预生成的 BCrypt 哈希（明文均为 "123456"，仅用于开发/测试）
-- 生产环境必须使用独立生成的随机哈希
UPDATE `user` SET password = '$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy';
UPDATE admin SET password = '$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy';
```

Admin 实体需添加 `@JsonIgnore`：

```java
@JsonIgnore
private String password;
```

AdminServiceImpl 需在保存管理员时调用 `passwordEncoder.encode()`。

### 4.2 手机号保护

#### 现状

| 表 | 字段 | 存储方式 | 风险评估 |
|----|------|----------|----------|
| `user` | `phone` | 明文 `VARCHAR(30)` | ⚠️ 数据库中直接可读 |
| API 响应 | `phone` | 无脱敏处理 | ⚠️ 前端展示完整手机号 |

#### 是否需要加密？

| 考量因素 | 分析 |
|----------|------|
| **数据敏感度** | 手机号属于《个人信息保护法》定义的个人信息，泄露可导致骚扰、诈骗 |
| **业务需要** | 买卖双方需要联系（面交），手机号是核心联系方式 |
| **行业惯例** | 主流平台对手机号做**脱敏展示**（如 `138****0001`）+ **数据库加密存储**（AES-256） |
| **本项目定位** | 校园二手交易，用户量有限，但需养成合规习惯 |

#### 建议方案

**方案一（推荐，平衡安全与复杂度）**：数据库明文 + API 脱敏展示

```
数据库：13800000001（明文）
API 响应：138****0001（脱敏）
仅订单交易双方可查看完整手机号
```

**方案二（高安全）**：数据库 AES 加密 + API 脱敏

```
数据库：AES_ENCRYPT('13800000001', secret_key)
API 响应：138****0001
后端解密仅在需要时执行
```

**建议**：当前阶段采用**方案一**（API 脱敏展示），后续升级到方案二。

API 脱敏实现示例：

```php
// 手机号脱敏工具方法
public static function maskPhone(?string $phone): ?string
{
    if (empty($phone) || strlen($phone) < 7) {
        return $phone;
    }
    return substr($phone, 0, 3) . '****' . substr($phone, -4);
}
```

### 4.3 邮箱保护

| 考量 | 分析 |
|------|------|
| 敏感度 | 低于手机号，但仍是个人信息 |
| 业务需要 | 注册/登录/通知，必须可读 |
| 建议 | 当前可明文存储。API 响应中可考虑部分脱敏展示（如 `a***e@example.com`），非必须 |

### 4.4 其他隐私考量

| 字段 | 敏感度 | 建议 |
|------|--------|------|
| `address` | 中 | 详细地址仅交易双方可见 |

## 五、补充发现

### 5.1 数据一致性

| 问题 | 详情 | 建议 |
|------|------|------|
| `data.sql` 密码与 BCrypt 不一致 | 种子数据存明文，代码期望 BCrypt 哈希 | 替换为预生成的 BCrypt 哈希 |
| `order_item.subtotal` 无约束 | `subtotal = unit_price × quantity` 依赖应用层保证 | 添加触发器或应用层校验 |

### 5.2 外键级联策略

当前所有 FK 均**未指定 ON DELETE / ON UPDATE** 行为，默认使用 InnoDB 的 `NO ACTION`（即 `RESTRICT`）：

| 场景 | 当前行为 | 建议 |
|------|----------|------|
| 删除用户 → 其发布的商品 | 被 FK 阻止删除 | 合理，商品有独立生命周期 |
| 删除商品 → 其图片 | 被 FK 阻止删除 | ⚠️ 建议 `ON DELETE CASCADE`，图片依附于商品 |
| 删除商品 → 收藏/评论/评分 | 被 FK 阻止删除 | ⚠️ 建议 `ON DELETE CASCADE` 或软删除 |
| 删除订单 → 订单明细 | 被 FK 阻止删除 | ⚠️ 建议 `ON DELETE CASCADE` |

建议修改：

```sql
-- 商品图片：商品删除时级联删除
ALTER TABLE good_image DROP FOREIGN KEY fk_good_image_good;
ALTER TABLE good_image ADD CONSTRAINT fk_good_image_good
    FOREIGN KEY (good_id) REFERENCES good (good_id) ON DELETE CASCADE;

-- 订单明细：订单删除时级联删除
ALTER TABLE order_item DROP FOREIGN KEY fk_order_item_order;
ALTER TABLE order_item ADD CONSTRAINT fk_order_item_order
    FOREIGN KEY (order_id) REFERENCES `order` (order_id) ON DELETE CASCADE;
```
