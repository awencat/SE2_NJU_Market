校园二手货交易平台需求规格说明书
================

* * *

1. 引言

-----

### 1.1 目的

本文档旨在定义“校园二手货交易平台”的功能与非功能需求，为后续设计、开发、测试提供依据。平台聚焦于学生之间闲置物品的交易，打造安全、便捷的校内二手市场。

### 1.2 范围

本系统面向在校大学生，提供二手物品发布、浏览、搜索、私信沟通、交易状态管理、评价等功能。支持的物品类型：书籍、电子产品、生活用品、体育用品、服饰等。  
本期范围不包括：在线支付（仅站内信约定面交或转账）、物流跟踪、跨校交易。

### 1.3 定义

* **卖家**：发布二手物品的学生用户。

* **买家**：浏览并有意购买物品的学生用户。

* **管理员**：审核违规内容、处理举报、管理用户的后台人员。

* **交易状态**：待售、已预订、已售、已下架。

* **信用分**：基于用户历史行为（交易完成率、被投诉次数）的评分，影响发布权限。

2. 总体描述

-------

### 2.1 产品视角

校园二手货交易平台是一个 web服务，让学生可以方便地出售闲置物品或购买所需物品，通过校内实名认证和信用体系降低交易风险。

### 2.2 用户特征

* **卖家**：希望快速清理闲置，获得一定收入，愿意拍照、定价、描述物品。

* **买家**：希望淘到低价好物，习惯搜索和筛选，关注物品新旧程度和价格。

* **管理员**：学校后勤或学生组织成员，需要处理纠纷和违规信息。

### 2.3 约束

* 开发周期：10周内完成可用版本（MVP）。

* 技术栈：springboot+vue+mysql。

* 认证方式：需对接学校统一身份认证系统（假设）。

* 内容限制：不得发布违禁品（如烟酒、电器等违反校规的物品），系统需提供敏感词过滤。

### 2.4 假设

* 用户愿意在校内面交或自行约定转账方式。

* 学生有拍照上传物品图片的条件。

* 用户会主动更新物品状态（如已售）。

3. 功能需求

-------

编号规则：`FR-模块-序号`。优先级：Must / Should / Could。

### 3.1 用户与认证模块

| 编号         | 需求描述                                 | 优先级    | 来源             |
| ---------- | ------------------------------------ | ------ | -------------- |
| FR-USER-01 | 系统支持授权一键登录。                          | Must   | AI生成           |
| FR-USER-02 | 新用户需补填手机号等个人信息。                      | Must   | AI生成+人工修改      |
| FR-USER-03 | 用户需实名认证（学号+姓名+学生证照片），管理员审核通过后才能发布物品。 | Must   | 人工补充（防止校外人员交易） |
| FR-USER-04 | 未认证用户只能浏览，不能发布或留言。                   | Must   | 人工补充           |
| FR-USER-05 | 用户可编辑各种资料（昵称、头像、宿舍区，以便面交）。           | Should | AI生成           |

### 3.2 商品发布与管理模块

| 编号          | 需求描述                                                          | 优先级    | 来源                             |
| ----------- | ------------------------------------------------------------- | ------ | ------------------------------ |
| FR-GOODS-01 | 已认证用户可以发布二手物品，必填：标题、类别、描述、原价、售价、新旧程度（全新/几乎全新/有使用痕迹）、图片（至少1张）。 | Must   | 人工补充（调研发现用户关注新旧程度）             |
| FR-GOODS-02 | 售价必须为0.01~5000元的数字，不能为负数。                                     | Must   | AI生成+人工修改（AI只写了“校验价格”，人工补充范围）  |
| FR-GOODS-03 | 发布时系统自动过滤违禁词（如烟酒、刀枪等），命中则提示“含有违规内容”。                          | Should | 人工补充（来自管理员需求）                  |
| FR-GOODS-04 | 系统应提供商品列表，按发布时间倒序展示。                                          | Must   | AI生成                           |
| FR-GOODS-05 | 支持按类别（书籍/电子产品/生活用品/体育用品/服饰/其他）筛选商品。                           | Must   | 人工补充                           |
| FR-GOODS-06 | 支持按价格区间（0-50, 50-100, 100+）筛选。                                | Should | 人工补充                           |
| FR-GOODS-07 | 支持按宿舍区筛选（方便面交）。                                               | Could  | 人工补充（调研中学生提到）                  |
| FR-GOODS-08 | 卖家可编辑或下架自己发布的商品（下架后不再展示）。                                     | Must   | AI生成                           |
| FR-GOODS-09 | 商品发布后7天未售出，系统自动提醒卖家“可以考虑降价或重新发布”。                             | Could  | AI生成+人工修改（AI原为“自动下架”，人工改为“提醒”） |
| FR-GOODS-10 | 每个卖家同时上架的商品数量不得超过20件。                                         | Should | 人工补充（防止刷屏）                     |

### 3.3 交易沟通模块

| 编号         | 需求描述                              | 优先级    | 来源          |
| ---------- | --------------------------------- | ------ | ----------- |
| FR-CHAT-01 | 买家可以在商品详情页点击“联系卖家”，发送站内私信。        | Must   | AI生成        |
| FR-CHAT-02 | 私信支持文字和图片（最多9张）。                  | Should | 人工补充        |
| FR-CHAT-03 | 卖家可以标记某个有意向的买家为“已预订”，商品状态变为“已预订”。 | Should | 人工补充（避免多人抢） |
| FR-CHAT-04 | 预订后24小时内若未完成交易，自动恢复为“待售”。         | Could  | 人工补充        |
| FR-CHAT-05 | 买家可收藏喜欢的商品，在“我的收藏”中查看。            | Should | AI生成        |

### 3.4 交易状态与完成模块

| 编号          | 需求描述                                  | 优先级    | 来源                  |
| ----------- | ------------------------------------- | ------ | ------------------- |
| FR-TRADE-01 | 卖家在商品售出后可将状态改为“已售”，商品从列表中隐藏但保留记录。     | Must   | AI生成                |
| FR-TRADE-02 | 买家收到物品后可以点击“确认收货”，但平台不涉及资金托管，仅作为信任标记。 | Could  | 人工补充（调研显示用户希望有确认环节） |
| FR-TRADE-03 | 双方在交易完成后可以互相评价（1-5星+文字）。              | Should | AI生成                |
| FR-TRADE-04 | 若发生纠纷，买家或卖家可发起“投诉”，管理员介入。             | Should | 人工补充                |

### 3.5 评价与信用模块

| 编号         | 需求描述                          | 优先级    | 来源                    |
| ---------- | ----------------------------- | ------ | --------------------- |
| FR-EVAL-01 | 交易完成后，双方可评价对方，评价后不可修改。        | Should | AI生成+人工修改（AI未写“不可修改”） |
| FR-EVAL-02 | 系统根据交易完成率、被投诉次数计算信用分（初始100分）。 | Should | 人工补充                  |
| FR-EVAL-03 | 信用分低于60分的用户不能发布商品，只能浏览和私信。    | Must   | 人工补充                  |
| FR-EVAL-04 | 用户可查看他人的信用分和好评率。              | Should | AI生成                  |

### 3.6 举报与审核模块

| 编号           | 需求描述                                  | 优先级  | 来源   |
| ------------ | ------------------------------------- | ---- | ---- |
| FR-REPORT-01 | 用户可举报违规商品或用户（售假、恶意骚扰等），填写举报原因。        | Must | AI生成 |
| FR-REPORT-02 | 管理员可查看举报列表，进行“通过”（下架商品/扣信用分/封禁）或“驳回”。 | Must | 人工补充 |
| FR-REPORT-03 | 被封禁用户无法登录和发布。                         | Must | AI生成 |

### 3.7 通知模块

| 编号         | 需求描述                                  | 优先级    | 来源   |
| ---------- | ------------------------------------- | ------ | ---- |
| FR-NOTI-01 | 系统在以下事件发送站内信：商品被留言、商品状态变为已预订/已售、投诉结果。 | Should | AI生成 |
| FR-NOTI-02 | 用户可开启微信订阅消息，接收实时通知。                   | Could  | 人工补充 |

4. 非功能需求

--------

| 编号                                      | 需求描述                         | 优先级    |
| --------------------------------------- | ---------------------------- | ------ |
| NFR-PERF-01                             | 商品列表加载时间 ≤ 3秒（4G网络）。         | Must   |
| NFR-PERF-02                             | 支持300人同时在线浏览，并发响应 ≤ 5秒。      | Should |
| NFR-SEC-01                              | 用户手机号仅在双方私信时可见，不可公开显示。       | Must   |
| NFR-SEC-02                              | 所有请求需校验登录态，防止越权操作。           | Must   |
| NFR-SEC-03                              | 图片上传需压缩，防止超大图片占用带宽。          | Should |
| NFR-USAB-02 | 发布商品时需有实时图片预览和删除功能。          | Should |
| NFR-MAIN-01                             | 关键操作（发布、下架、举报、封禁）记录日志，保留30天。 | Should |
| NFR-MAIN-02                             | 后台可导出商品、用户、举报列表（Excel）。      | Could  |
