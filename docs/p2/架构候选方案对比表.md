# 任务 1：架构风格选择与论证

## 1.1 候选架构分析

本项目为“校园二手交易平台”，主要面向校内学生提供二手商品发布、浏览、搜索、收藏、沟通、订单管理、评价等功能。考虑到本项目周期较短，团队开发经验有限，且系统规模属于课程项目级别，因此架构选择需要重点关注：开发效率、可维护性、10 周内可落地性，以及后续功能扩展空间。

本阶段主要比较以下两种候选架构方案：

- **方案 A：分层架构 + MVC 模式的单体应用**
- **方案 B：微服务架构**
- **方案 C：事件驱动架构**

### 候选方案说明

#### 方案 A：分层架构 + MVC 模式的单体应用

该方案将系统划分为表现层、业务逻辑层、数据访问层和数据库层。前端负责页面展示和用户交互，后端按照 Controller、Service、Repository/Mapper 的方式组织代码。

典型结构如下：

```text
前端页面
  ↓
Controller 层：接收请求、参数校验、返回响应
  ↓
Service 层：处理业务逻辑
  ↓
Repository/Mapper 层：访问数据库
  ↓
数据库
```

这种方案适合中小型 Web 系统，结构清晰，开发和部署成本较低。

#### 方案 B：微服务架构

该方案将系统拆分为多个独立服务，例如用户服务、商品服务、订单服务、消息服务、评价服务等。每个服务独立开发、独立部署，并通过 HTTP API 或消息队列进行通信。

典型结构如下：

```
前端
  ↓
API 网关
  ↓
用户服务 / 商品服务 / 订单服务 / 消息服务 / 评价服务
  ↓
各自数据库或共享数据库
```

这种方案适合业务规模较大、团队分工明确、后期需要高扩展性的系统。

#### 方案 C：事件驱动架构

该方案强调通过事件进行模块之间的解耦。例如用户下单后发布“订单创建事件”，通知消息模块、库存模块、通知模块等进行后续处理。

典型结构如下：

```
用户操作
  ↓
业务模块产生事件
  ↓
消息队列 / 事件总线
  ↓
其他模块异步消费事件
```

这种方案适合复杂业务流程、异步任务较多、对模块解耦要求较高的系统。

## 初步结论

综合比较后，本项目更适合采用：

> **分层架构 + MVC 模式的单体应用**

原因如下：

1. **符合项目规模**
    校园二手交易平台虽然包含用户、商品、订单、聊天、评价等模块，但整体仍属于中小型系统，不需要一开始就采用复杂的微服务架构。
2. **开发成本可控**
    课程项目周期较短，团队总工时有限。分层架构能够让团队快速完成核心功能，而不是把大量时间消耗在服务拆分、服务治理和部署配置上。
3. **便于团队分工**
    前端、后端、数据库设计、测试文档可以相对独立开展。后端内部也可以按模块划分，如用户模块、商品模块、订单模块、消息模块等。
4. **可维护性较好**
    通过 Controller、Service、Repository/Mapper 的分层方式，可以避免业务逻辑全部堆积在接口层，使代码结构更清晰。
5. **后续仍有扩展空间**
    虽然当前选择单体架构，但可以在模块划分时保持清晰边界。未来如果系统规模扩大，可以逐步将商品、订单、消息等模块拆分为独立服务。

# 1.2 关键实验：“架构辩论赛”

## 实验目的

本实验旨在比较不同 Prompt 策略对 AI 架构建议质量的影响。团队成员分别采用不同方式向 AI 咨询架构方案，并记录 AI 的回答，再通过团队讨论选择最终架构。

------

## 成员 A：直接询问 AI 推荐架构

### Prompt

```
我们要做一个校园二手交易平台，请推荐一个合适的软件架构。
```

### AI 回答摘要

AI 建议可以采用 MVC 架构或前后端分离架构。理由是校园二手交易平台包含用户注册登录、商品发布、商品浏览、订单交易、评价等功能，MVC 可以将页面展示、业务逻辑和数据访问分开，便于开发和维护。

AI 同时提到，如果项目未来规模扩大，可以考虑微服务架构，将用户服务、商品服务、订单服务、消息服务拆分出来。

### 回答质量分析

优点：

- 能够快速给出常见架构建议。
- 提到了 MVC 和微服务两种可能方案。
- 方向基本正确。

不足：

- 对项目约束理解不足。
- 没有明确考虑课程项目周期、团队人数和实现难度。
- 对“为什么不选微服务”分析较浅。
- 没有给出具体模块边界。

### 评分

| 维度         | 评分 |
| ------------ | ---- |
| 相关性       | 4    |
| 可落地性     | 3    |
| 深度         | 2    |
| 对约束的考虑 | 2    |
| 综合评分     | 3    |

## 成员 B：提供详细项目约束后询问 AI

### Prompt

```
我们正在做一个课程项目：校园二手交易平台。项目周期约 10 周，团队 4 人，主要功能包括用户登录、商品发布、商品浏览与搜索、收藏、订单管理、交易评价和站内消息。团队成员对前后端开发有基础，但没有完整微服务开发经验。请结合开发复杂度、可维护性、可扩展性、性能、团队学习成本和 10 周内可行性，推荐合适的软件架构。
```

### AI 回答摘要

AI 推荐采用“前后端分离 + 后端分层架构 + MVC 模式”的单体应用。AI 认为该方案更适合 10 周课程项目，因为它复杂度较低、便于分工、可以快速实现核心功能，同时后端仍可按照用户模块、商品模块、订单模块、消息模块进行清晰划分。

AI 不建议直接使用微服务架构，因为微服务会引入服务注册、API 网关、服务通信、分布式事务、部署和监控等额外问题，对课程项目来说成本较高。

AI 建议可以在单体架构中预留模块边界，未来需要时再逐步拆分为微服务。

### 回答质量分析

优点：

- 充分考虑了项目周期、团队人数和技术经验。
- 明确推荐了适合课程项目的架构。
- 对不选择微服务给出了较充分理由。
- 给出了可演进的设计思路。

不足：

- 对具体模块接口说明不足。
- 对性能和数据一致性问题讨论较少。
- 对消息通知、聊天等功能是否需要异步机制分析不够深入。

### 评分

| 维度         | 评分 |
| ------------ | ---- |
| 相关性       | 5    |
| 可落地性     | 5    |
| 深度         | 4    |
| 对约束的考虑 | 5    |
| 综合评分     | 5    |

## 成员 C：要求 AI 对比多种方案优劣

### Prompt

```
请针对校园二手交易平台，对 MVC 分层架构、微服务架构、事件驱动架构进行对比分析。请从开发复杂度、可维护性、可扩展性、性能、团队学习成本、10 周内可行性几个维度进行比较，并给出最终推荐方案。
```

### AI 回答摘要

AI 对三种架构进行了横向比较：

- MVC 分层架构：开发简单、结构清晰、适合中小型系统。
- 微服务架构：扩展性强，但开发和部署复杂度高。
- 事件驱动架构：适合异步通知和模块解耦，但对初学团队来说理解和调试成本较高。

AI 最终推荐使用 MVC 分层架构作为主体架构，并建议在部分功能中吸收事件驱动思想，例如订单创建后触发消息通知、评价提醒等。

### 回答质量分析

优点：

- 横向比较较完整。
- 能够从多个质量属性进行分析。
- 指出事件驱动可以作为局部辅助，而不是整体主架构。
- 最终建议较合理。

不足：

- 对“校园二手交易平台”的具体业务场景结合还不够深入。
- 表格中的部分描述偏通用。
- 没有明确说明如何在代码层面组织模块。

## 成员 D：让 AI 扮演架构师角色进行分析

### Prompt

```
你是一名软件架构师。我们要在 10 周内完成一个校园二手交易平台，团队 4 人，目标是完成可运行原型和架构文档。系统功能包括用户认证、商品管理、搜索、收藏、订单、评价、消息通知。请你从架构师角度分析应该选择什么架构，并说明关键权衡、风险和后续演进方案。
```

### AI 回答摘要

AI 从架构师视角建议采用“前后端分离 + 模块化单体 + 分层架构”。AI 认为项目当前最重要的目标不是支撑高并发，而是保证核心功能可实现、代码结构清晰、便于测试和交付。

AI 给出的关键权衡包括：

- 放弃微服务带来的独立扩展性，换取更低的开发和部署复杂度。
- 采用模块化单体，保证当前开发效率，同时为未来拆分服务保留可能。
- 暂不引入复杂消息队列，避免增加环境配置和调试难度。
- 对订单、商品、消息等模块保持清晰接口，降低后续维护成本。

AI 还指出风险包括：如果模块边界不清晰，单体项目后期容易变成“大泥球”；如果业务逻辑写在 Controller 中，会导致维护困难；如果数据库表设计不合理，后续订单和评价功能扩展会受影响。

### 回答质量分析

优点：

- 回答更加贴近架构设计文档的写法。
- 不仅给出方案，还分析了权衡和风险。
- 强调“模块化单体”，比简单 MVC 更准确。
- 能够区分当前目标和未来演进目标。

不足：

- 仍需要人工结合课程要求整理成正式文档。
- 对技术选型没有直接给出完整表格。
- 对模块接口的数据格式说明不足。

| 相关性       | 5    |
| 可落地性     | 5    |
| 深度         | 5    |
| 对约束的考虑 | 5    |
